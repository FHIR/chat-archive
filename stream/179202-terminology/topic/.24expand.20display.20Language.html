Ted Klein (Jun 04 2017 at 22:12):

IMHO (but this should be further discussed), an unprofiled expand should return what is in the definition (the display in <compose> is part of the VSD), plus what is requested by the API operation (if it exists; a language may be requested in an expand on an intensional definition where some of the expanded concepts do not have displays in the requested language). An expand profile should have be ability to specify if the expansion should contain the concept code and only the requested display. So I'm suggesting that the default is to give what is in the VSD if specified in an extensional PLUS what is requested in the argument, but make sure a profile can tighten that on request.
