We just released a draft set of validation tests for the slot publishing actor, and deployed them on our qa/test site. Robert Scanlon (Apr 09 2021 at 19:52):

A quick way to try it out is plug in https://raw.githubusercontent.com/smart-on-fhir/smart-scheduling-links/master/examples/ as the 'FHIR endpoint', and leave the two input fields blank when you run the tests. That will download and validate all of the example resources.

Yeah, Inferno as a platform has a FHIR RESTful API bias, which is why it expects the 'base FHIR url' to be the starting point. But I agree in the context of this IG, the manifest itself is the obvious starting point to test publishers, and breaking if the user pastes that into the front box is silly. I just pushed up an update that will work if you paste in a direct link to the manifest in the front-page box (e.g. https://raw.githubusercontent.com/smart-on-fhir/smart-scheduling-links/master/examples/$bulk-publish). Future versions of Inferno will likely remove the front page input box.

Robert Scanlon (Apr 11 2021 at 04:05):

Is this formally built on top of FHIR's RESTful API, and if so, should I add a capabilities interaction (/metadata) test? But on the other hand, support for the capabilities interaction is a hard requirement in FHIR for a reason.
