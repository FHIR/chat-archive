David Hay (Oct 08 2020 at 18:33):

Have made the changes discussed yesterday. Specifically:

  • 'product' is required. When a product is entered previous surveys for that product are shown in a separate tab. If the survey has been entered through the same browser, then there is a delete option. This seemed the simplest way to allow updates/corrections as well as a way of checking that no-one else has completed the survey for a given product.
  • UI changes to the location of the filter options (by resource type or previously selected)
  • any number of FHIR versions can be selected - and changed to "Lloyd order". The resources listed are derived from the selected versions.
  • 'results so far' tab lists contactable numbers (where an email has been entered) - though not the actual emails of course
  • if the call is HTTP, it will redirect to HTTPS

I didn't do the 'upload capability statement' option as mentioned in previous posts - but could look at that. Probably the easiest would be for the user to specify the root of the server, and have the app go fetch it. The server would need to support CORS of course, and the metadata endpoint unprotected...

IG's implemented is another interesting one. I'd suggest it's a different component of the survey rather than a way of finding supported resources. As @John Moehrke says, it would be elegant to get that from the capability statement, though you'd want a manual option as well. We'd also want to have a picklist for IG's - though I guess we'd also want to allow manual entry. Supported IG's may be of wider interest than supported resources too...

Comments welcome - and testing would be much appreciated! (It works for me :) ).
