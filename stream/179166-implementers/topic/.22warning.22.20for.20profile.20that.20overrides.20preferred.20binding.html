Topic: "warning" for profile that overrides preferred binding

Pieter Edelman (Mar 08 2021 at 15:43):

I have this R4 profile on PractitionerRole, which overrides the core ValueSet binding to c80-practice-codes on PractionerRole.specialtywith a custom ValueSet. This base binding is "preferred", so it shouldn't be a problem to replace it.
However, when I try to validate an example instance with a code from my custom ValueSet, I still get the following message:

None of the codes provided are in the value set http://hl7.org/fhir/ValueSet/c80-practice-codes (http://hl7.org/fhir/ValueSet/c80-practice-codes), and a code is recommended to come from this value set)

This message is just of level "information" but still it bugs me. Shouldn't the derived profile override the base ValueSet? So why is it still recommended to use the base ValueSet?

Lloyd McKenzie (Mar 08 2021 at 15:46):

The validator validates against both the profile and the base resource - the warning stems from validation against the base resource.

Pieter Edelman (Mar 08 2021 at 16:22):

Ho @Lloyd McKenzie , I get that. I just wonder why. If I interpret the specs correctly, the base binding shouldn't be considered at all if the derived profile overrides this binding.

Lloyd McKenzie (Mar 08 2021 at 16:27):

Profiles can't 'override' the base bindings - the base bindings still hold. So even if you're using a profile that declares a required binding, the preferred binding for the base resource is still applicable. (And implementers should, ideally, satisfy both). The only situation we might want to suppress a based binding is if the base is a preferred binding and is on a non-repeating Coding element.
