Topic: ✔ FHIR Type for CDS Feedback

Oliver Cardoza (Sep 08 2021 at 13:23):

Hello everyone, apologies for the long post but I believe it's better to be thorough :)

I'd like to solicit recommendations on the most suitable FHIR type for clinical decision support (CDS) "Feedback". The "Feedback" concept is outlined in the current draft of CDS-Hooks. The FHIR R4 specification already includes recommendations to structure other CDS concepts within GuidanceResponse and RequestGroup. Full article: 14.5 Clinical Decision Support Services.

The purpose of CDS Feedback is for a user (e.g. Practitioner) to send feedback on a recommendation (e.g. GuidanceResponse) made by a CDS service (e.g. Device) back to said service. This positive/negative/neutral feedback can help the CDS service improve.

I've thought of a few options for consideration but I'm also interested in other ideas I may have missed.

A) Use Communication type (preferred): This type can fit the desired properties nicely because it identifies a sender - Ref(Patient|Practitioner|...) sending info to a recipient - Ref(Device|...)). It can also refer to the recommendation which the feedback is responding to using about - Ref(Any). However, it does not have a great place to structure the CDS Feedback outcome which would need to be an extension.

B) Extend GuidanceResponse type: The feedback could be embedded within it's parent GuidanceResponse as an extension. I'm not a fan of this approach because I believe feedback should be structured as a separate type.

C) Extend Basic type: The feedback would be structured as a new type by extending Basic. It can use the subject - Ref(Any) to identify the GuidanceResponse and the author - Ref(Patient|Practitioner|...) to identify the user providing feedback. Extensions would then be used to identify the CDS Feedback outcome and more (e.g. recipient device, send time).

Appreciate all thoughts, ideas, and feedback. And wow you made it to the end, you deserve a treat! Maybe go grab a cappuccino, you deserve it!

Lloyd McKenzie (Sep 08 2021 at 19:29):

It's mainly a question of "where should this data be surfaced". A system that's managing and tracking Communications related to a patient (i.e. what data has been shared about them) would be somewhat surprised to see CDS Feedback entries showing in that space. On the other hand, a system that's managing GuidanceResponses should be able to tolerate/understand that information better.

Lloyd McKenzie (Sep 09 2021 at 15:11):

Communication isn't really about 'transmission', it's about a record of what was shared. If you hand the patient a pamphlet, that's Communication. If you want to record that you faxed a lab result to the patient's GP because they weren't on the recipient list, that's a Communication.
