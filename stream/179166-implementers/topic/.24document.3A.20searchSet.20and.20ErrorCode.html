Patrick Werner (Jan 13 2019 at 01:01):

http://build.fhir.org/composition-operation-document.html states that
The server takes the composition resource, locates all the referenced resources and other additional resources as configured or requested and either returns a full document bundle, or returns an error. Note that since this is a search operation, the document bundle is wrapped inside the search bundle.

Today in a discussion with @Alexander Zautke we were not quite sure what error should be returned, and when.
We assumed that the error is returned if the composition is not found, so -> 404 and probably when some problems with including resources linked from the composition resource are occurring. -> 500 ??
The second issue we are having is the fact that the document shall be bundled inside a searchSet bundle. This complicates the processing of the document bundle, as i first have to unwrap the SearchSet before you can persist your bundle somewhere else. What is the benefit of the SearchSet Bundle around the Document Bundle, and could we get rid of it?

The $document is a useful operation for many usecases. It could be referenced in a DocumentReference similarly to XDS concept of On-Demand. That is that there is a DocumentReference that describes the ability to create a certain kind of On-Demand document, where the DocumentReference.content.attachment.url is the service endpoint for that kind of document $document operation.
