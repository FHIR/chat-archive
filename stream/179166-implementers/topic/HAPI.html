Lloyd McKenzie (Oct 28 2020 at 15:28):

All security/access-based filters will get tied into the back end. There are a few architectural approaches.

  1. You can execute all inbound queries against a database 'view' that enforces the access privileges a particular user (in a given context) has.
  2. You can take the SQL (or other) syntax for the inbound query and append it to the SQL/other syntax for the user's permissions such that the returned data is the intersection of the two.
  3. You can execute the query and then perform a post-filter to strip out the results the user doesn't have the right to see (or suppress the data elements within that set they're not allowed to see)

These mechanisms can be combined, and there can potentially be other approaches too. Each of the reference implementations has layers where you ought to be able to invoke these approaches.
