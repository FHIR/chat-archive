Topic: _include=*

Paul Church (Aug 26 2021 at 16:03):

This occurred to me from the points @Josh Mandel made in the discussion on https://chat.fhir.org/#narrow/stream/204607-CARIN-IG.20for.20Blue.20Button.C2.AE/topic/_include.3A*

In the spec, the definition of _include=* is "indicating that any search parameter of type=reference be included". This wording is a bit imprecise and assumes that the list of search parameters is identical to the list of search parameters supported for _include.

The CARIN BB IG is unusual in that it requires a bunch of _includes without requiring the ability to search by those parameters - but one can also imagine the opposite case where a search parameter is supported for search but not _include.

Should we clarify this wording to "indicating that any search parameter of type=reference that is supported for include/revinclude be included"?

Josh Mandel (Aug 26 2021 at 17:36):

Yes please!

Josh Mandel (Aug 26 2021 at 17:38):

Although for CARIN it's not clear how a server could advertise that it supports certain parameters for _include without supporting them for search; this clarification would enable the converse only, right?

Paul Church (Aug 26 2021 at 17:45):

The clarification enables both but doesn't address how you would advertise it. Over in the cap statement you could list the include names in rest.searchInclude without their definitions, but that doesn't seem sufficient.

nicola (RIO/SS) (Aug 26 2021 at 23:31):

For performance and security reasons we decided do not implement _include=* in Aidbox. As in math logic, all universal quantifiers are dangerous :) I would vote for the exclusion of this feature.

Corey Spears (Aug 27 2021 at 04:53):

@Josh Mandel , Forgive me if I am either stating the obvious or just missing something here, but the CapabilityStatement.rest.resource.searchInclude specifies the _include values supported by the server. The other FHIR guidance on the meaning of _include (https://www.hl7.org/fhir/search.html#revinclude) aside, this is not explicitly linked to the CapabilityStatement.rest.resource.searchParam elements.

The CARIN group was thinking about explicitly requiring those searchInclude values, explain that the _include=* meant the minimum of those specified (though a server could add more). The means for which to implement that would be left up to the server (one option being they could add those references as search parameters).

Any concerns about this approach?
