Kyle Thompson (Dec 02 2021 at 02:34):

Consider a Task with restriction.period set.

I was reading through the Task State Machine documentation (https://www.hl7.org/fhir/task.html#statemachine) and was wondering, if a Task is in progress and a user doesn't complete the Task in the specified period, it looks like the Task should transition to the "failed" state. In the documentation for Task statuses (https://www.hl7.org/fhir/valueset-task-status.html), it defines failed to mean "The task was attempted but could not be completed due to some error". I am hung up on "..due to some error". Would not completing it in the specified period, not because of error but because the owner just wasn't able to complete the Task in time, counts as an error or is an error here meant more as a system error?

I realize this Task State Machine is a "typical" example, but I'm curious if the desired behaviour in my example is to transition the Task to a failed state or if "error" here means something stricter than how I am interpreting it.

Lloyd McKenzie (Dec 02 2021 at 03:03):

I think we've got a change request that refines that description somewhat in R5. "Some error" might better be framed as "some problem". It could be system, user, environment or anything else. The real world intervened in a way that means "this isn't going to get finished".

Lloyd McKenzie (Dec 02 2021 at 03:04):

So "didn't do it in the necessary time, so it's either not going to happen, or if it does, it won't matter/count" would definitely fall into the "failed" bucket.

Kyle Thompson (Dec 02 2021 at 14:28):

OK, rephrasing "some error" to "some problem" makes sense. Thanks!
