Topic: _count > server page size

Doug Pratt (Sep 11 2020 at 12:34):

If a client app specifies a _count value higher than the server page size (say 75 vs. 50), the spec clearly allows the server to return 50, rather than 75. But the spec also states: The server should repeat the original _count parameter in its returned page links so that subsequent paging requests honor the original _count.

I believe that the intent here is that the client can use the URL to get the next page without having to append the parameter in code for every subsequent request. In the use case where _count is greater than the server page size, should the server return a count reflecting its page size rather than repeating the original count (i.e., 50 rather than 75)? Either would technically work, but returning 50 seems more useful and aligned with the intent.

René Spronk (Sep 11 2020 at 13:25):

The 'self' link reflects the parts of the search URL as understood /executed by the server. If the client indicated 'max 75 responses', then a server that uses 50 certainly meets that requirement. In that sense _count-75 is certainly true. _count is a max number, not a precise page size.

René Spronk (Sep 11 2020 at 13:27):

@Lloyd McKenzie One could argue either way. The current wording as cited by Doug gives clear guidance, as such there's no ambiguity.

Thanks guys, it's great to hear from you! How would you feel about amending the paragraph as follows:
The server should repeat the original _count parameter in its returned page links so that subsequent paging requests honor the original _count.
