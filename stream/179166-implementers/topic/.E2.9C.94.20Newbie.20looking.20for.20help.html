Daniel Venton (Sep 01 2021 at 13:00):

All of these specifications have one thing in common, they are attempting to define interoperability, or how to standardize the transmission of healthcare data.
HL7 FHIR defines a collection of resources, this is how an "observation" is defined. These are the attributes that go along with it etc.
HL7 FHIR also defines a restful API that defines how to query and push data around, using the above resources.

IG's are meant for certain use cases.
US Core, does your exchange of information have to comply with USCDI? If so you'll want to pay attention to this.
CARIN BB, does you exchange of information have to comply with CMS Patient Access API? If so this IG applies.
PDEX formulary, CMS Patient Access API.
PDEX Plannet, CMS Patient Access API.
You are required to comply with an IG because a 3rd party requires its use.

IHE is much like an HL7 IG (my experience with them, was IHE transactions and CDA documents), but most of it still applies. There are certain interactions that, if you follow them, allows you to connect (hopefully) seamlessly with other software applications with no direct testing effort.

Much like how lightbulb manufacturers don't have to test with every lamp maker, both actors follow the standard and can prove they follow the standard, thus every lightbulb should work, out of the box, with every lamp. Every HL7 FHIR compliant server should work out of the box, with every HL7 FHIR compliant client application.

Lloyd McKenzie (Sep 01 2021 at 15:00):

The base FHIR standard is quite wide open - almost all elements are optional and many have very loose expectations around what codes are used, etc. Implementation guides tighten down that base specification for use in a particular space. For example, in Patient, name is optional - because in some use-cases, name might not apply (Patient is a chicken) or not be allowed to be shared (anonymized public health reporting). However, in other use-cases, having name is really important. Implementation guides take the base standard and tighten it down to what's relevant in a narrower space - a particular country, a particular type of patient, a particular set of use-cases. As a result, you get better interoperability within that space.
