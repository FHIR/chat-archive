Topic: stable/on-demand in MHD

René Spronk (Nov 22 2019 at 07:11):

The on-demand option in XDS.b states the following when it comes to the ITI-18 stored query: "If no value is specified for DocumentEntryType, the value requesting only Stable Document Entries shall be assumed. To get all Document Entry types, the query shall contain both of the valid values in the request.". In MHD we can't distinguish (by means of a query parameter), whether something is a stable or an on demand document. Wouldn't it be better if the default in XDS.b were to be 'both', rather than 'stable'? (raised by Revathy Sreenath, at the Amsterdam DevDays)

John Moehrke (Nov 22 2019 at 15:57):

It would have been better, but IHE governance prevents breaking changes to a Final Text profile. The concept of dynamic documents was added much later, after the core XDS was normative. Thus it had to be added as an Option.

John Moehrke (Nov 22 2019 at 15:57):

very little down side to the model where you need to ask for both if you want both.

John Moehrke (Nov 22 2019 at 15:59):

If the main point of the question is how dynamic documents are to be addressed in MHD... That topic has not been brought to IHE as something needing to be solved. The original scope for MHD was kept small to be easy to write and implement. Thus some XDS options were not addressed, including dynamic (onDemand) documents.

René Spronk (Nov 23 2019 at 08:58):

The question is a) an observation that we have a difference in default query behaviour between XDS.b and NHD, where personally I think the XDS.b wording is the undesirable one, and b) a query as to what (if any) is the use case where one would wish to explicitely only retrieve on-demand or stable documents?

John Moehrke (Nov 25 2019 at 15:24):

Not much to say on ( a ). But on ( b ) there is little overhead to returning results that the client throws away. I have no heard any requests for ( b ) either

It is certainly reasonable to want an IPS to represent the right-now summary of information about a subject of care. Is this a use case we could add to the MHD HIE work item?
