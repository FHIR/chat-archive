Github Notifications (FHIRcast) (Nov 26 2019 at 15:59):

isaacvetter opened PR #303 from wss-in-content-location to master:

During an II call last week, Bill Wallace pointed out that it would be both classier and more compatible with DICOM's use of websockets to return the wss url in the Content-Location HTTP header, instead of in the body of the response.

Personally, I think this is a bit more RESTful -- we get to re-use some HTTP and is likely a small change to existing prototypes and implementations. We ought to hear feedback from implementers to confirm this is a reasonable change.
