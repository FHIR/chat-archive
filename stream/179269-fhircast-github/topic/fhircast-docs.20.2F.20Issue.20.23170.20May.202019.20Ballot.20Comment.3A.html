Github Notifications (FHIRcast) (May 07 2019 at 04:05):

isaacvetter commented on Issue #170

Great point, Yunwei. This id is typically the notification identifier that the subscriber receives. But in the "Request Context Change Request", the subscriber is proposing a context change which has not yet occurred and, in fact, may not occur!

I think that it would be better for the subscriber to not submit an id for the context change at all, and rather, (if and) once accepted, the hub should assign an id upon broadcasting the event notification to all subscribers. (This is similar to a FHIR create, in which the newly created resource id is assigned by the server, not the client). Do note that the original requestor also receives the notification request once it's enacted --

Once a requested context change is accepted, the Hub MUST broadcast the context notification to all subscribers, including the original requestor.

Proposed Resolution: Persuasive with Mod
Remove the id element from the example "Request Context Change Request" message and clarify that the there's no id in the subscriber's request.

Github Notifications (FHIRcast) (May 09 2019 at 14:26):

isaacvetter commented on Issue #170

Resolution: Persuasive

Clarify that the id element generated by a subscriber and supplied in the "Request Context Change Request" as follows is supplied in the request, generated by the subscriber, and potentially won't be echoed by the hub following an accepted context change.

Should re-use this language from #247 :

This id SHALL be unique for the Hub, for example a GUID.

Elliot Silver moved the following disposition, seconded by Reinhard Egelkraut

Disposition: Persuasive
Disposition Comment: Will clarify that the ID for subscribers requesting a context change SHALL be generated by the subscriber, and will follow the same wording as proposed in issue resolution #247. Refer to Isaac's comment above.
