Not sure if this is the correct stream, but here we go.

We are looking to create a set of IGs to publish as packages. The IGs will be fairly small, one per domain, in order to have separate release-pipelines, versioning and dependencies.

From the registry doc.:

DIY: Create your own package feed and add it to the list of package feeds:
- Create an RSS feed (template at http://hl7.org/fhir/package-feed.xml)
- Register the package feed in https://github.com/FHIR/ig-registry/blob/master/fhir-ig-list.json (by making a GitHub PullRequest. The package -server will regularly check the RSS feed for new packages.

The IG development is done on GitHub and new releases are created using GitHub's release management including changelog and NPM Package.

The current plan is to create a github-action to trigger on release events and generate a XML according to the Fhir package RSS feed template. This xml is hosted on github and can be registered as a package feed in the fhir-ig-list.json as mentioned in the registry doc.

Such a solution should integrate well in our current CI\CD pipeline. However, I'm concerned after finding only 11 registered feeds in the fhir-ig-list since this indicates that creating your own feed is not a common practice. I'm also curious about the package-restrictions and guides lists because they are not mentioned in the registry doc., do we need to populate them as well?
