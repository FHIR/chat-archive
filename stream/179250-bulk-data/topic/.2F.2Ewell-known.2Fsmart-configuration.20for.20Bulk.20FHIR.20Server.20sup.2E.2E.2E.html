This is what I have cooking for DPC thus far:

{
  "response_types_supported": [
    "token"
  ],
  "id_token_signing_alg_values_supported": [
    "RS384"
  ],
  "token_endpoint": "https://sandbox.cms.gov/api/v1/Token/auth",
  "grant_types_supported": "client_credentials",
  "scopes_supported" : "system/*.*",
  "token_endpoint_auth_methods_supported": ["private_key_jwt"],
  "claims_supported": [
    "iss",
    "sub",
    "aud",
    "iat",
    "exp",
    "jti"
  ],
  "capabilities": ["backend-services"]
}

It seems that smart backend would require a few more parameters like:
"client_assertion_type":"urn:ietf:params:oauth:client-assertion-type:jwt-bearer" that the openid config spec doesn't have

DPC also has a single /Token/auth endpoint. This endpoint takes a JWT (which the initial client id obtained from the DPC web as JWT issuer and subj) and returns a bearer access_token authorizing the user to use the API. This could also be construed as the token_endpoint or the authorization_endpoint or both.?

I have reviewed the HL7 example here (http://www.hl7.org/fhir/smart-app-launch/conformance/index.html#using-well-known), and the open id connect discovery spec, and it seems it may need some further work for SMART backend.
