Jeffrey Taylor (Sep 04 2019 at 20:14):

On page 32 of the HL7 FHIR Bulk Data API.pdf presentation, there is a note about adding components like de-identification that act as nodes in the $export process. Is there any ideas floating around on how one might request an extra 'pipeline' item (like de-id) to run on the output (or during?) an $export operation?

Jeffrey Taylor (Sep 04 2019 at 20:15):

HL7 FHIR Bulk Data API.pdf i'm speaking of can be found here: https://www.hl7.org/documentcenter/public_temp_EAFAF532-1C23-BA17-0C9A3E480C937714/calendarofevents/himss/2018/HL7%20FHIR%20Bulk%20Data%20API.pdf

Josh Mandel (Sep 04 2019 at 20:26):

Nothing like that today; the idea is that any pipeline node might expose its own $export endpoint. For example, organization might deploy a bulk data server behind a de-id proxy, and tell clients about the de-id proxy endpoint (or about both, and let the clients choose which endpoint to call).

Jeffrey Taylor (Sep 04 2019 at 20:30):

I see. In this case, how would a request to de-id proxy endpoint differ than the regular endpoint? Would this just be something in the DNS specifying which endpoint is de-id and which is not? Or is there an opportunity here to say something like.. /Patient/$export?_someParam=some-de-id-operation-specifier? Where this would do both the bulk export & de-id of results.

Josh Mandel (Sep 04 2019 at 20:32):

The request would be a standard export request (just issued against a server that happens to return de-id results... and as a further implementation detail, a server that happens to be implanted as a proxy, sitting in this kind of pipeline arrangement in front of a fully identified export server).

Jeffrey Taylor (Sep 04 2019 at 20:34):

Makes sense. Thank you for the clarification.
