Topic: _until parameter

Vladimir Ignatov (Jun 01 2021 at 13:15):

The _since parameter should limit the results to resources modified since the given date. In most cases the client would do that if it already has the previous data, which provides basic synchronization capabilities.

However, this implies that:

  1. Client's first export was full (no _since)
  2. The client "remembers" the date of the last export and passes it as _since parameter to the next export

Making the first (full) export may be expensive and not always desirable. If we pair the _since parameter with another one called _until (or _before or whatever the best name is), then it would be possible to create clients that can "fill in the gaps" in the data that they already have.

For example, if a client already has the data for 2019-2020 but wants to have everything starting from 2018

  • Currently we have to pass _since=2018 which works but re-exports the entire 2019
  • Alternatively, we could split that to 2 exports and omit 2019: _since=2018&_until=2019 and _since=2020

Also, that could eliminate the need to make that huge first export, potentially reducing the load on servers.

Dan Gottlieb (Jun 01 2021 at 19:22):

@Vladimir Ignatov thanks for bringing this up! IIRC, including _until was discussed on one of the Argonaut calls and the group ultimately concluded that while there are certainly use cases where it would be helpful (e.g., the rough pagination you mentioned, filling in a failed historical request), it didn't seem critical enough to most of the core bulk use cases to be worth expanding the number of params included in the IG.

Dan Gottlieb (Jun 01 2021 at 19:23):

That said, even if it’s not written into the spec now, since the IG allows additional parameters to be used in the kickoff request, servers that do have a strong use case to support _until can add it and test it out for possible inclusion in future versions of the IG.
