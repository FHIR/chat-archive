@Nancy Lush , @Peter Jordan , I understand that you demoed your patient summary solution at today's GDHP session. I wasn't able to attend, and only heard about it second hand, but it raised some questions for me (and it ties in with something I was asking about recently).

If someone asks for a summary, you're allowed to return an existing one, but how do you determine that there is an existing one? In your system, can you have a document for someone other than a locally known patient, and if so what would happen if I tried to ask for that document to be returned in a summary? (I know you can't actually ask for a document to be returned, but I mean invoke the operation in such a way that that existing document would otherwise be returned).

Peter Jordan (Jan 26 2022 at 20:17):

It's worth reading this new page in the IPS IG. Whether there is a pre-existing summary on a server, or it will attempt to create one 'on-the-fly' will depend on individual system requirements. Suffice to say that healthcare data is event-driven and pre-build summaries can quickly go stale.

Elliot Silver (Jan 26 2022 at 20:22):

Using a $summary operation on the Patient resource. ... This operation is similar to how the $everything operation works on the Patient resource.

So, by my read, there must be a Patient resource (at /Patient) that this is being invoked on; and having a patient that exists only in an existing document bundle isn't sufficient.

Peter Jordan (Jan 26 2022 at 20:34):

That's not quite how I read it. AFAIA, a server can hold an IPS instance without 'having' (i.e. persisting) all of the individual resources in the Bundle (i.e. the resource references can point to another server) - e.g. if it is an aggregator that takes information from other servers. In that case, it may 'have' a patient resource for the purposes of creating the Bundle, but might not persist it thereafter. The aim here would be to reduce the number of copies of data in a particular eco-system.

John D'Amore (Jan 27 2022 at 17:28):

@Elliot Silver Here's two workflows for consideration:
1) If a server has stored an IPS Bundle, then a DocumentReference could be created to point to that Bundle. Using search on DocumentReference or a $docref operation, you may be able to find the IPS reference even if all data haven't been persisted as discrete resources.

2) If the Patient resource exists, then using $summary would allow you to create an IPS from information in that system using the rules that system has for creating a patient summary (so long as it meet minimum IPS requirements).

I do agree with @Peter Jordan that summaries can go stale quickly. In the US, C-CDAs run into this concern quite readily. While the answer isn't always to make things "on-demand" that often is regarded as the safest option. On demand C-CDA generation with FHIR is something supported in the US through $docref operation: https://www.hl7.org/fhir/us/core/OperationDefinition-docref.html

Welcome feedback on this topic and the new page (http://build.fhir.org/ig/HL7/fhir-ips/ipsGeneration.html) referenced by Peter.
