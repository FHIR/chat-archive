Hank Lenzi (Jan 14 2022 at 12:15):

Hello --

When gofsh processes files, it looks at the "name:" in the JSON file. This will produce the fish file with that "name". However, when you run the produced file through sushi, it'll complain about the files having to be ASCII encoded.

Granted, this is but a minor annoyance for anybody working with Latin-1 alphabets (ISO/IEC 8859-1 - sushi-ascii-centrism2.png the people with the ás, the és, the ças, the ås, äs, æs etc), but this is too anglo-centric, IMHO, if I dare say so. Like I said, for Latin-1 alphabets, that's a minor nuisance - but what about non-romanized alphabets?

Not only that, this means more JSON tweaking, and hence more opportunity for bugs. Is this really something that is necessary? In 2021, the toolchain for many languages already have embraced UTF-8 encodings (e.g., Rust etc.). sushi-ascii-centrism2.png

Chris Moesel (Jan 14 2022 at 14:26):

Hi @Hank Lenzi. I understand your concern, but SUSHI is just following the recommendations and constraints established by the FHIR specification itself. If you look at how FHIR defines ValueSet.name, you'll see this:
image.png

The bit outlined in red is saying that a name should start with A-Z and consist only of A-Z, a-z, 0-9, and _. Anything else should produce a warning (which is what SUSHI does). But again... this was established by the FHIR spec, not SUSHI or FSH.

If you want SUSHI to suppress these warnings, you should be able to do so via a sushi-ignoreWarnings.txt file, documented here.

Lloyd McKenzie (Jan 20 2022 at 02:24):

Name is intended to be used for computation purposes only (class names, table names, etc.) The rules were established to ensure that names would work with most/all such software tools and languages. (I'll admit that many of these suffer from being overly presumptive around the use of the U.S. character set.) If you want to convey something intended for reading and use by humans, use 'title' - and you're free to use any valid Unicode characters you like there.
