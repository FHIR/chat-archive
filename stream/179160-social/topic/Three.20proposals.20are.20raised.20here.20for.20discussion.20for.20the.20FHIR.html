1 . The security label can be applied to the elements’ layer rather than resources.
For example, the consent of the owner of the health information only applies to the part of the CONDITION resource instead of the entire resource.

  1. A separate triage tag can be specified out of the meta.tags for priority and time sensitivity of patient safety resource information. This triage tag can enable the system to push the information to the registered endpoints instead of waiting for the pull.

David Pyke (Apr 05 2022 at 17:36):

  1. That was reviewed and rejected as unneeded by the large majority of the community by the FHIR team
  2. That can be added by creating a custom profile. If you wish to see it added to the Core spec, you can add a ticket in jira.hl7.org
  3. If you wish to see it added to the Core spec, you can add a ticket in jira.hl7.org with suggested language

Lloyd McKenzie (Apr 05 2022 at 20:21):

With respect to #3, it's hard to define guidance that applies universally because there is such variation in architecture, requirements and expectations. For some systems and types of exchanges, multiple messages an hour is overload. For others, less than 100/second is a sign some of their participants are probably having network issues. Guidance about things like "number of layers" almost always belongs in an implementation guide where the implementation guide can establish a shared set of expectations in which establishing such architectural expectations is reasonable/realistic.
