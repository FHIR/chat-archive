David Pyke (Jul 29 2021 at 19:24):

FHIR is an interoperability specification for healthcare data. Oversimplified, SMART on FHIR is a OAuth specification that handles authentication and authorization for FHIR servers.

Lloyd McKenzie (Jul 29 2021 at 20:36):

Detailed discussion on SMART on FHIR can also happen here: #smart. Essentially SMART on FHIR leverages FHIR for data access, but adds an additional framework on top of it to manage app launch, permissions, identity verification and authorization.

But some examples have a different value other than this set. Look at this example (http://hl7.org/fhir/R4/encounter-example-f203-20130311.json.html). How is this possible? Thanks

Morten Ernebjerg (Sep 02 2021 at 09:03):

Hi & welcome :wave: (BTW questions are best asked in the #implementers stream). Your are right that Encounter.type has a binding to the value set you mention. However, the binding only has the strength "example". That means that there is no requirement (or even expectation) that instances will use codes from this set - it is merely way of indicating what kind of codes might be used in this element. To enforce the use of certain codes in an element, one needs a required or extensible binding - see the documentation of binding strengths.

Lloyd McKenzie (Jan 03 2022 at 04:29):

@Yash Giradkar There are a variety of ways to contribute. You can provide feedback on the core specification or most implementation guides using the 'propose a change' link at the bottom of each page. You can decide to get involved with one or more of the work groups that help define and maintain the content (the responsible work group is a link on most pages in the core spec and generally in the footer of implementation guides. You can get involved in supporting one or more of the reference implementations or other tools the community has created. And of course, you can go forth and create tools of your own. So your first decision is: how do you want to get involved? :) Welcome.

Lloyd McKenzie (Jan 12 2022 at 19:23):

Using the publication machinery for the core spec probably wouldn't work very well. There's a whole lot of code that is specific to certain resources, data types, etc. and it would be difficult to make it work. However, you could probably create a publication that looked quite similar using the ImplementationGuide publisher tool and define all of your proprietary models as FHIR logical models. We don't currently generate diagrams for logical models, but that's something we could (and probably should) add.
