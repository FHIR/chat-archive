Simone Heckmann (Jan 07 2019 at 16:46):

Vorschlag zur Beantwortung der Frage:

Grundsätzlich gilt:
Jedes CodeSystem/ValueSet kann n Identifier haben, aber immer nur eine (canonical) url.
In der Instanz eines Codes wird immer die CodeSystem-url angegeben.
Jeder Code existiert daher innerhalb eines Namensraumes immer nur einmal mit einem eindeutigen Namespace.

Die OIDs werden als alternative Identifier der ValueSets/CodeSystems mitgeführt, um diese anhand der OID auffindbar zu machen, was insbesondere zur Implementierung von Mappings (V3/CDA->FHIR) erforderlich ist.

Es ist theoretisch möglich, eine OID als canonical -url eines CodeSystems zu verwenden, in der Notation urn:oid:1.2.3.4.5.6, Grundsätzlich bemüht sich das TC jedoch, sprechende URLs zu verwenden, die idealerweise - aber nicht zwingend - auch zur Definition des jeweiligen Artefaktes auflösen.
In manchen Fällen kann dies jedoch nicht möglich sein, da die Terminologien von Autoritäten publiziert werden, auf die wir keinen Einfluss haben. Wenn es uns nicht möglich ist, eine geeignete URL mit den Autoren abzustimmen, wird im Notfall auch die OID als CodeSystem-URL verwendet. Das sollte jedoch die absolute Ausnahme sein.

Aus meiner Sicht sollte das auch die Präferenz/Empfehlung sein, selbst dann wenn die Nutzung der von der zuständigen Autorität vergebenen ID dann nicht zu einer sprechenden URI führt.

Simone Heckmann (May 08 2019 at 08:00):

Mail ist raus!
Zur Info an alle, dies ist die Zusammenfassung der Anforderungen, die wir dem DIMDI übermittelt haben:

[...]Canonical URLs in FHIR sind zunächst lediglich universell eindeutige Bezeichner eines Conformance-Artefaktes
(z.B. eines ValueSets, CodeSystems, oder einer NamingSystem).
Referenz: http://hl7.org/implement/standards/fhir/references.html#canonical
In so fern sind sie als Synonyme einer OID zu betrachten, die auch häufig nach einem ähnlichen Schema benannt werden.
Ausgehend von der Basis-Domäne einer Organisation (z.B. http://meine-organisation.de) werden Zweige für
unterschiedliche Arten von Artefakten vergeben.
z.B.:
http://meine-organisation.de/ValueSet/
http://meine-organisation.de/CodeSystem/
http://meine-organisation.de/NamingSystem/
Die Artefakte selbst erhalten dann einen eindeutigen Bezeichner.
z.B.:
http://meine-organisation.de/ValueSet/aufnahmearten
http://meine-organisation.de/NamingSystem/patienten-identifier

Im Prinzip würde es also ausreichen, die CanonicalURL als alternativen Bezeichner einer OID in einem zusätzlichen Feld im Register hinterlegen zu können.
Allerdings haben (canonical) URLs den Vorteil, dass die direkt zur Definition eines Artefaktes auflösen können.
Entwickler erwarten dies im Allgemeinen auch, da es das Auffinden der Definition erheblich erleichtert.

Wenn z.B. die canonicalURL http://meine-organisation.de/NamingSystem/patienten-identifier
bei der Eingabe in einen Browser über einen automatischen Redirect auf die URL
https://portal.dimdi.de/websearch/servlet/Gate?accessid=showOidDoc&query=url=http://meine-organisation.de/NamingSystem/patienten-identifier
weitergeleitet würde, benötigen die Anwender keine Kenntnis über die Adresse der Portalseite, um zur Spezifikation gelangen zu können.

Diese Möglichkeit sollte unbedingt geschaffen werden.
Dazu ist es jedoch erforderlich, dass der Seitenaufruf nicht nur über den oid-Parameter sondern alternativ auch über einen (canonical)url-Parameter (siehe Beispiel) erfolgen kann.

Die URL unter "tables and schemas" wiederum ist ein physischer Link auf eine Internetseite oder ein Dokument,
das das Artefakt genauer spezifiziert und ist von der Canonical URL unabhängig.

Für die Implementierung von Übersetzungen zwischen CDA und FHIR wäre es wichtig, dass eine Mapping-Tabelle von sämtlichen OIDs nach CanoncialURLs und umgekehrt zum Download bereitgestellt wird.
Dies kann bspw. als CSV und/oder FHIR ConceptMap erfolgen.
