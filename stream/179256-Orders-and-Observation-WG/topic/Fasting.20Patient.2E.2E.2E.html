Eric Haas (Apr 26 2018 at 15:36):

  • OPTION 1 : based on the LRI Guide ( V2) is OBR-13. This would correspond to either a new extension or element DiagnosticReport.fastingStatus.

  • OPTION 2: a stand alone Observation that is referenced in ServiceRequest.supportingInfo when the diagnostic is requested on already existing specimens. If the request is entered first with an unknown specimen, The Specimen resource references to the ServiceRequest. Create a new supportingInfo element on Specimen where the patient fasting status Observation could be represented. Other stuff like specimen condition could go here as well ( or in notes like is commonly done today :-)). (note there is a pending tracker on DiagnosticReport to add a supportingInfo element which could also reference the Observation on the reporting end)

Created a Logical FHIR Model on ClinFhir to demonstrate: http://clinfhir.com/logicalModeller.html#ge21z

  • OPTION 3: A fastingStatus and condition elements on Specimen.

Created a Logical FHIR Model on ClinFhir to demonstrate: http://clinfhir.com/logicalModeller.html#96vh7

  • OPTION 4: to split Specimen (i.e., the tube of blood) from the collection (e.g., Procedure) not sure how the fasting status would be represented here??? but specimen condition would still be an element on Specimen as in OPTION 3,
